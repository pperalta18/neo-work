# Music Sync (Beats) Implementation Plan

## Summary

Make Remotion compositions choreographable against a song. A track is analysed
**once, offline** (essentia.js, WASM) into a deterministic **beat map** (BPM,
beats, downbeats, onset accents/*golpes*, sections), stored as fps-agnostic JSON
in `public/audio/`. The composition author reads a **frame-indexed summary** to
know exactly which frame each beat/accent lands on, queries the map with pure
helpers, and locks motion to it via `<AudioTrack>` runtime hooks — all derived
from `useCurrentFrame()` so renders stay deterministic ("state in, frame out").

Key decisions (settled in the design discussion):
- **essentia.js** is the analysis engine, confined to the offline script — it
  never enters the Remotion bundle or the MP4. The render path reads JSON only.
- Beat-map times are in **seconds**; `frame = round(t * fps)`. Same song, any fps.
- The map is committed and **hand-editable**; auto-detection is a starting point.
- No visual editor/overlay — the readable beat map *is* the authoring surface.
- `ProductTourVideo` becomes the beat-driven reference (scene length in bars,
  transitions fire on downbeats), replacing today's hardcoded `SCENE_FRAMES=95`.

See `specs/music-sync.md` for the behavioural spec.

## Phase 1: Beat-map contract & authoring helpers

- [x] Create `src/lib/beatmap.ts` with the `BeatMap` type: `duration`, `bpm`,
      `beats[]`, `downbeats[]`, `onsets[]`, `sections[]` (`{name,start,end}`),
      optional `bands` (`{hz, low[], mid[], high[]}`) — all times in seconds.
- [x] Add `secondsToFrame(t, fps)` (`Math.round(t * fps)`) and the inverse
      (`frameToSeconds(frame, fps)`).
- [x] Add pure authoring helpers: `beatFrames(map, fps)`, `accentFrames(map, fps)`
      (onsets), `downbeatFrames(map, fps)`, `nextDownbeat(map, frame, fps)`,
      `sectionAt(map, frame, fps)`.
- [x] Add `formatFrameSummary(map, fps)` → the human/agent-readable string
      (beats / downbeats / golpes / sections lines) reused by the CLI.
- [x] Add a hand-authored fixture map (`src/lib/__fixtures__/intro.beats.json`)
      and Vitest unit tests for every helper (boundaries, empty sections, fps 24/30/60).

## Phase 2: Offline analyser (`npm run beats`)

- [x] Add `essentia.js` as a **devDependency** (used only by the script).
- [x] Create `scripts/analyze-beats.mjs`: decode an audio file to mono PCM (ffmpeg),
      run essentia.js (`RhythmExtractor2013`) for BPM + beat positions; derive
      downbeats by 4/4 bar grouping; onset accents via `OnsetRate`; section
      boundaries by phrase grouping. (Band envelopes left optional/unused for now;
      the pure derivations live in `src/lib/beatAnalysis.ts` so they unit-test
      without booting WASM, with the `.mjs` as the ffmpeg+essentia shell.)
- [x] Write the result to `<audio>.beats.json` conforming to the `BeatMap` schema
      (`validateBeatMap` asserts sorted/in-range events + contiguous sections
      before writing).
- [x] Print `formatFrameSummary` to stdout (reused from `beatmap.ts`); support
      `--fps <n>` (default 30) and `--bpm <n>` / `--offset <s>` manual overrides,
      plus `--bar` / `--section-bars` / `--out`.
- [x] Add `"beats": "node scripts/analyze-beats.mjs"` to `package.json` scripts.
- [x] Drop a test track into `public/audio/` (`test-beat.wav`, a synthesized
      120 BPM 4/4 click track generated by `scripts/make-test-tone.mjs`) and
      generate its committed `test-beat.beats.json`; detection lands on ~120 BPM
      with beats every 15 frames @30fps — sanity-checked against the known grid.

## Verification

- [x] All Vitest unit tests for `beatmap.ts` helpers pass.
- [x] **Determinism**: rendering ProductTour twice yields byte-identical frames.
      Pure `determinismReport` + `chooseSampleFrames` (`src/lib/verifyDeterminism.ts`)
      — sampling picks endpoints + the scene-cut frames (where spring/transition
      drift would surface) + even coverage; the verdict FAILS on any hash mismatch
      AND on an empty sample set (a gate that rendered nothing must not pass).
      `scripts/verify-determinism.mjs` (`npm run verify:determinism`) bundles the
      entry (mirroring remotion.config.ts), renders each sampled frame TWICE to
      separate PNGs, sha256s them, and hands the pairs to the pure report.
      Unit-tested in `src/lib/verifyDeterminism.test.ts` (13 tests, incl. a
      positive control that a hand-built mismatch is reported — not a no-op).
      Confirmed live: ProductTour (1920×1080 @30fps, 240f) sampled at
      [0,60,75,120,135,179,239] (75/135 = the downbeat cuts) → all 7 byte-identical
      across two renders, exit 0.
- [x] **No essentia.js in the bundle**: essentia.js is a **devDependency** (not a
      runtime dep), and the Remotion bundle contains no WASM analysis code. Pure
      `scanBundleForForbidden` (`src/lib/verifyBundleClean.ts`) keys on essentia's
      code surface (`EssentiaWASM`/`RhythmExtractor2013`/`OnsetRate`) + any `.wasm`
      asset — deliberately NOT the bare word "essentia" so doc comments don't
      false-positive. `scripts/verify-bundle-clean.mjs` (`npm run verify:bundle`)
      bundles the entry via `@remotion/bundler` (mirroring remotion.config.ts's
      alias/rules), walks the output, and scans it. Unit-tested in
      `src/lib/verifyBundleClean.test.ts` (11 tests). Confirmed live: real bundle =
      67 files, 0 `.wasm`, `[OK]` exit 0; positive-control proves the walk+scan
      actually detects markers (not a no-op).
- [x] The `npm run beats` frame summary matches the JSON (`secondsToFrame` of each
      beat equals the printed frame at the chosen fps) — `formatFrameSummary`
      prints `beatFrames`/`downbeatFrames` of the same map; asserted in
      `analyze-beats.test.ts` against the committed `test-beat.beats.json`.
- [x] The rendered ProductTour MP4 contains an audio track, and scene cuts land on
      downbeats when previewed against the music. Audio: `verify:tour-audio` (AAC
      stereo ≈8.04s muxed). Cuts-on-downbeats is now checked against the **rendered
      pixels** (not just layout math) by pure `analyzeCutTiming` + `changeSignalFromGray`
      (`src/lib/verifyCutTiming.ts`) + `scripts/verify-cut-timing.mjs`
      (`npm run verify:cuts`): ffmpeg decodes the MP4 to a 64×36 grayscale change
      signal, `planBeatScenes` (the same layout the composition renders) gives the
      expected downbeat cuts + transition length, and the gate asserts the transition
      energy is concentrated on those cuts. The verdict is dynamic-range-normalised so
      a soft fade and a bold slide are judged alike, and the deliberate caption
      `useBeatPulse` accents on non-cut downbeats are distinguished from scene
      transitions (not flagged). Unit-tested in `src/lib/verifyCutTiming.test.ts`
      (18 tests: positive, eased/front-loaded, missing/misplaced/off-centre/smeared
      negatives, stray-equal-size, accent-not-transition, end-to-end raw frames).
      Confirmed LIVE: ProductTour → [OK] exit 0 (90% of transition energy on cuts
      [75,135]; cut 75 prominence 1.00 centroid off 1.7f; cut 135 prominence 0.35 off
      3.3f; the only stray, the 195 caption pulse, correctly below transition scale).
      Negative control LIVE: out/conversation.mp4 → [FAIL] exit 1 (concentration 0.5%,
      both cut windows empty) — proves the pipeline discriminates, not a no-op. The
      only residue is the purely subjective "feels synced when watched", which is
      inherently human; every objective part is now gated.
- [x] Same beat map renders correctly at 24 / 30 / 60 fps (no hardcoded fps in
      helpers). Pure `fpsInvarianceReport` (`src/lib/verifyFps.ts`) lays the tour
      out at each fps and asserts (a) each rate spans exactly `secondsToFrame(dur,
      fps)` with `renderedLength == totalFrames`, (b) every cut sits on
      `round(downbeatSeconds·fps)` re-derived from the map, and (c) all fps land
      cuts on the **same downbeat indices** (the fps-independent fingerprint) —
      cross-checking the rates against each other, which the per-fps layout tests
      can't. `scripts/verify-fps.mjs` (`npm run verify:fps`) prints per-fps frame
      summaries + verdict and, with `--render`, bundles + renders one ProductTour
      still per fps (fps/duration overridden) to prove the React tree builds.
      Unit-tested in `src/lib/verifyFps.test.ts` (15 tests, incl. a positive
      control that `crossFpsConsistent` rejects fps that disagree — not a no-op).
      Confirmed live: `--render` produced 24/30/60fps stills (cuts [60,108]/[75,135]/
      [150,270], indices [1,2]), exit 0. (Needed explicit `.ts` extensions on the
      Node-loaded runtime chain `verifyFps.ts`→`beatScenes.ts`→`beatmap.ts`.)
